一、项目概述

1.1 项目背景

企业内线电话作为日常办公的基础通信资源，涉及申请、分配、使用、回收、变更等多个环节。当前各子公司/部门的号码资源分散管理，依赖手工 Excel 台账和线下流程，存在以下痛点：
  • 资源不透明：号码分配回收全靠人工记录，容易出现重复分配、遗漏回收的情况，资源利用率无法量化。
  • 流程效率低：号码申请、设备变更、人员调动等操作需要跨部门协调，纸质或即时工具流转，耗时且易丢失。
  • 费用分摊难：运营商账单到手后需要人工拆分到各成本中心，工作量大且容易出错，缺乏自动化对账能力。
  • 数据孤岛：组织架构、员工信息、号码资源、设备台账分散在不同系统或文件中，无法联动查询。
项目定位：构建统一的企业内线电话管理平台（PhoneBiz），实现号码全生命周期数字化管理、费用精准自动分摊、多维度报表自动生成，提升管理效率和数据透明度。

1.2 项目目标

序号目标度量指标
1号码全生命周期管理：覆盖申请→分配→使用→回收→变更全流程号码状态100%可追踪，操作全流程线上化
2费用自动分摊：运营商账单自动导入，按规则分摊到成本中心分摊准确率≥99%，人工干预减少80%
3多维度报表：按组织、号码类型、费用维度自动生成报表报表生成时间从天级降到分钟级
4工单驱动：关键号码操作通过工单驱动，可追溯、可审批操作100%有工单记录，审批SLA≤2工作日
5多角色协作：运维、财务、管理层各取所需，权限分明4类角色覆盖全部业务场景

二、业务需求

2.1 组织架构管理

业务价值：组织架构是整个系统的基础数据，号码分配、费用分摊、员工关联均依赖组织。树形结构支持多级子公司/部门，确保资源归属清晰。

功能清单

序号功能描述优先级角色
1组织创建创建组织节点，支持设置组织名称、组织编码、上级组织、负责人、联系电话、区号等基本信息P0ADMIN
2组织编辑修改组织基本信息，组织编码创建后不可修改（作为唯一标识）P0ADMIN
3组织删除/停用删除无关联数据的叶子组织节点；非叶子节点或有关联数据时仅支持停用P0ADMIN
4树形结构展示以树形列表展示组织层级，支持展开/折叠、搜索过滤P0ADMIN, OPS, FINANCE, BOSS
5组织编码规则自动生成组织编码（如 ORG-001），支持手动指定前缀P1ADMIN

业务规则

  • 组织编码全局唯一，创建后不可修改
  • 树形深度不超过 6 级（支持1级集团→2级子公司→3级事业部→4级部门→5级科室→6级班组）
  • 删除组织前必须检查是否存在关联员工、号码、设备；有关联时拒绝删除并提示具体关联数量
  • 组织停用后，其下属所有子组织一并停用，不影响已有号码的正常使用
  • 每个组织可配置默认区号，用于该组织下号码的前缀拼接

前置条件

无外部依赖，作为基础模块首先实施。

用例描述

UC-ORG-001 创建组织：ADMIN 进入组织管理页面，点击"新增组织"按钮，在弹出表单中选择上级组织（默认为根节点），填写组织名称、组织编码（可留空自动生成）、负责人姓名、联系电话、默认区号，提交后系统校验编码唯一性，创建成功后在树形列表中展示新节点。

UC-ORG-002 停用组织：ADMIN 在组织列表中选择目标组织，点击"停用"按钮，系统检查该组织下是否有活跃员工或分配中的号码，若有关联则弹出提示"该组织下有 X 名员工、Y 个活跃号码，确认停用？"。确认后组织及其子组织状态变更为停用，前端以灰色样式展示。

2.2 员工管理

业务价值：员工是号码使用者和费用承担者的关联实体。通过员工与组织的归属关系，实现"人→组织→号码→费用"的完整链路追踪。

功能清单

序号功能描述优先级角色
1员工创建录入员工基本信息：姓名、工号、手机号、邮箱、所属组织、岗位、入职日期P0ADMIN, OPS
2员工编辑修改员工信息，工号创建后不可修改P0ADMIN, OPS
3员工删除/离职离职时标记状态，自动触发关联号码回收流程P0ADMIN, OPS
4员工查询按姓名、工号、手机号、所属组织等条件搜索，支持分页P0ADMIN, OPS, FINANCE, BOSS
5员工批量导入通过 CSV/Excel 批量导入员工，支持模板下载和校验反馈P1ADMIN

业务规则

  • 工号全局唯一，创建后不可修改
  • 手机号格式校验：11位中国大陆手机号（1[3-9]\d{9}）
  • 员工必须关联一个有效（未停用）的组织
  • 离职操作：员工状态变更为"离职"，若该员工名下有已分配或活跃状态的号码，自动创建回收工单
  • 组织间调动：变更所属组织时，不自动变更号码归属，需单独发起号码变更流程

前置条件

依赖组织架构管理模块（2.1），需先创建目标组织。

用例描述

UC-EMP-001 员工离职处理：OPS 在员工管理页面选中目标员工，点击"离职"按钮，系统弹出确认框显示"该员工名下有 X 个活跃号码，离职后将自动创建号码回收工单"。确认后员工状态变更为离职，系统自动为每个活跃号码创建类型为"号码回收"的工单，初始状态为"待处理"，工单描述自动生成"员工[姓名]离职，号码[号码]需回收"。

2.3 号码全生命周期管理

业务价值：号码管理是系统的核心模块，覆盖号码从录入到回收的完整生命周期。通过状态机管理、并发控制和工单驱动，确保号码资源的安全和可追溯。

2.3.1 号码基础管理

号码是系统的核心资源实体，每条号码记录包含号码值、类型、状态、归属组织、关联员工、关联设备等信息。号码状态通过状态机严格管理，确保每个状态转换都有明确的触发条件和操作记录。

号码6种状态：

状态英文标识说明允许操作
空闲IDLE已录入系统但未分配给任何员工分配、冻结、删除
已分配ALLOCATED已分配给员工但尚未激活使用激活、冻结、回收
活跃ACTIVE正在使用中的号码冻结、变更、回收
冻结FROZEN因异常或管理需要临时冻结，暂停服务解冻、回收
待回收PENDING_RECOVERY已发起回收但尚未完成（等待设备注销、话费结清等）确认回收、取消回收
已回收RECOVERED已回收至号码池，可重新分配重新分配、归档

功能清单

序号功能描述优先级角色
1号码录入手动录入号码，填写号码值、类型（外线/内线/分机）、运营商、所属组织、区号P0OPS
2号码查询按号码值、状态、类型、组织、员工等条件组合查询，支持分页和导出P0ADMIN, OPS, FINANCE, BOSS
3号码详情查看号码完整信息，包括操作历史、关联工单、关联设备、费用记录P0ADMIN, OPS
4号码删除仅允许删除"空闲"状态且无操作记录的号码P1OPS
5号码统计按状态、类型、组织维度统计号码数量和占比P1ADMIN, OPS, BOSS

业务规则

  • 号码值在系统内全局唯一（同一号码值不可重复录入）
  • 号码类型：OUTER（外线）、INNER（内线）、EXTENSION（分机）
  • 状态转换必须符合状态机规则，非法转换请求返回 400 错误
  • 每次状态变更记录到号码操作历史表（phone_number_history），含操作人、时间、原状态、新状态、备注
  • 号码归属组织变更需通过工单流程

前置条件

依赖组织架构管理（2.1），号码必须归属于某个组织。

2.3.2 分机号池管理

业务价值：分机号资源有限且复用率高，通过区间管理+用量预警，避免分机号冲突和资源耗尽，保障业务连续性。

功能清单

序号功能描述优先级角色
1号池区间创建定义分机号区间（如 8001-8100），关联组织，设置最大容量P0OPS
2号池用量监控实时显示区间内已用/可用数量，三色预警（绿<70% / 黄70-90% / 红>90%）P0OPS, ADMIN
3号池区间编辑修改区间名称、关联组织、扩展区间范围P1OPS
4号池区间回收释放不再使用的号池区间，释放前需确认区间内号码已全部回收P1OPS
5自动分配分机号从号池中自动选取下一个可用分机号分配给员工P0OPS

业务规则

  • 号池区间不可交叉重叠，如已有 8001-8100 则不可创建 8050-8150
  • 自动分配策略：按号池内最小可用号码优先分配（从小到大）
  • 用量超过 90% 时触发系统告警，通知运维人员
  • 号池关联组织后，该号池仅服务于该组织及下属组织的分机号需求

2.3.3 区号对照

业务价值：不同地区的分公司使用不同区号，通过组织级区号映射，确保号码拨打和费用归集的准确性。

功能清单

序号功能描述优先级角色
1区号配置为组织配置默认区号（如 010、021、0755），支持组织级覆盖P1ADMIN
2区号查询查看组织的区号配置，支持按区号反查组织P1ADMIN, OPS
3标准区号库内置全国标准区号库，录入时自动校验区号合法性P2—

业务规则

  • 区号必须符合国家通信标准格式（以 0 开头，2-4位数字）
  • 子组织未单独配置区号时，继承上级组织的区号
  • 区号变更不影响已有号码记录（历史数据保持不变）

2.3.4 号码分配与回收

关键特性：号码分配是系统的高并发场景，必须保证幂等性和防重复分配。使用数据库唯一约束 + 乐观锁双重保障。

功能清单

序号功能描述优先级角色
1手动分配号码OPS 选择空闲号码分配给指定员工，可指定组织、备注P0OPS
2自动分配号码指定号码类型和组织，系统自动从可用池中选取号码分配P0OPS
3号码回收将活跃/已分配/冻结号码回收至空闲状态，支持创建回收工单P0OPS
4批量回收按组织、员工、批量选择号码一键回收P1OPS
5回收前置检查回收前检查是否有未结清费用、未完成工单，有则提示确认P0OPS

业务规则

  • 分配操作通过 UPDATE ... WHERE status = 'IDLE' 行级锁保证并发安全
  • 分配接口幂等：相同请求（员工+号码）重复提交返回已分配结果而非报错
  • 一个员工同一时间最多持有 5 个活跃号码
  • 回收时自动解除号码与设备、分机号的关联
  • 回收操作写入号码操作历史，记录回收原因和操作人

前置条件

依赖员工管理（2.2）和号码基础管理（2.3.1）。

2.3.5 号码状态变更

功能清单

序号功能描述优先级角色
1号码冻结将活跃/已分配号码冻结，暂停通话服务，记录冻结原因P0OPS
2号码解冻将冻结号码恢复为冻结前的状态（活跃或已分配）P0OPS
3发起回收将号码状态变更为"待回收"，等待后续确认P0OPS
4确认回收确认待回收号码完成回收，状态变更为"已回收"P0OPS
5取消回收取消待回收状态，恢复为原状态P1OPS

业务规则

  • 冻结原因分类：异常话费（疑似盗打）、员工违规、管理要求、设备故障
  • 冻结超过 30 天未解冻，系统自动创建预警工单
  • 待回收状态超过 15 天未确认，系统发送提醒通知
  • 所有状态变更必须填写备注说明原因

2.3.6 号码变更（携号）

高风险操作：号码变更涉及"断开旧号码→分配过渡号→激活新号码→回收过渡号"四个步骤，使用双锁机制确保数据一致性。

功能清单

序号功能描述优先级角色
1号码变更申请申请将员工A的号码X变更为号码Y（携号），说明变更原因P0OPS
2过渡号自动分配变更期间自动分配一个临时过渡号，确保通信不中断P0系统自动
3变更执行执行四步变更流程，通过双锁（旧号码锁+新号码锁）保证一致性P0系统自动
4变更回滚变更失败时自动回滚到变更前状态P0系统自动

业务规则

  • 变更流程：旧号码冻结 → 分配过渡号 → 新号码激活 → 旧号码回收 → 过渡号回收
  • 使用数据库事务 + SELECT ... FOR UPDATE 对旧号码和新号码加行锁，防止并发冲突
  • 任何步骤失败自动回滚已执行的操作，恢复到变更前状态
  • 变更全程记录到号码操作历史，每一步都有独立记录
  • 目标号码（新号码）必须处于"空闲"状态

2.4 话机与设备管理

2.4.1 话机基础管理

业务价值：话机是内线电话的物理终端，通过管理设备的型号、MAC地址、注册状态、关联号码，实现"号码-设备-人"的端到端可追溯。

功能清单

序号功能描述优先级角色
1设备录入录入话机信息：设备名称、型号、MAC地址、SN序列号、所属组织、安装位置P0OPS
2设备编辑修改设备信息，MAC地址和SN创建后不可修改P0OPS
3设备删除仅允许删除未关联号码的设备P1OPS
4设备查询按MAC地址、型号、组织、关联号码、注册状态查询P0ADMIN, OPS
5号码关联将设备与号码关联/解除关联，一个设备同一时间仅关联一个号码P0OPS

业务规则

  • MAC地址全局唯一，格式校验：^[0-9A-Fa-f]{2}(:[0-9A-Fa-f]{2}){5}$ 或 ^[0-9A-Fa-f]{12}$
  • SN序列号全局唯一
  • 设备状态：ONLINE（在线）、OFFLINE（离线）、UNREGISTERED（未注册）、FAULT（故障）
  • 删除设备前必须解除与号码的关联

2.4.2 话机操作

组织（Organization） → 树形自关联，支持多级组织层级
员工（Employee） → 多对一关联组织（一个组织下多个员工）
号码（PhoneNumber） → 多对一关联组织，多对一关联员工（可选），多对一关联设备（可选）
设备（PhoneDevice） → 多对一关联组织，多对一关联号码（可选）
工单（WorkOrder） → 多对一关联创建人，多对一关联审批人，多对多关联号码
账单（Bill） → 多对一关联成本中心，一对多关联账单明细（BillItem）
成本中心（CostCenter） → 多对一关联组织
发票（Invoice） → 多对一关联账单，多对一关联成本中心

关键关联说明：

  • 号码和员工是可选关联：空闲状态的号码不关联员工，已分配/活跃状态必须关联员工
  • 号码和设备是可选关联：并非所有号码都有物理设备（如虚拟分机号）
  • 工单与号码是多对多：一个工单可涉及多个号码（如批量回收），一个号码可出现在多个工单（如先申请后回收）
  • 账单明细与号码是一对一：每条账单明细对应一个具体号码的当月费用
