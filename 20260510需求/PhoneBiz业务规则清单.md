# PhoneBiz 业务规则清单 v1.0

> 整理：AI 辅助 | 日期：2026-05-08 | 状态：待确认

## 一、组织与人员规则

### ORG-001：组织层级
- 集团为根节点，下设子公司和部门，层级不限
- 支持一个子公司下再设子公司（多级子公司）
- 组织类型：`group` / `subsidiary` / `dept`

### ORG-002：Admin 管理范围
- Admin（行政）用户的管理范围按组织结构树定义
- 可管理本节点及下级所有节点（部门/子部门）
- 具体范围由系统管理员在用户创建时分配

### ORG-003：员工与用户绑定
- 每个员工对应一个系统用户账号（绑定同一工号）
- 员工离职后账号状态变为 `inactive`，不影响号码历史数据

### ORG-004：工号规则
- 工号统一为 6 位字符
- 纯数字工号：直接作为分机号（去掉前导 0，分机号不以 0 开头）
- 含字母工号：系统从分机号池中随机分配一个 6 位数字分机号
- 分机号池由系统管理员按部门分配，各部门独立分机号资源池

## 二、电话号码规则

### PHONE-001：号码全局唯一性
- 电话号码（phone_number）在全系统全局唯一
- 分机号（extension_number）在全系统全局唯一
- 每个员工同一时间最多持有 1 个电话号码 + 1 个分机号

### PHONE-002：号码状态定义

| 状态值 | 含义 | 可否分配 |
|--------|------|----------|
| idle | 空闲/未分配 | ✅ 可直接分配 |
| active | 在用/正常 | - |
| stopped | 停机/暂停 | 需先复机 |
| cancelled | 已拆机 | ❌ 不可恢复 |

### PHONE-003：共享号码（is_shared）
- 共享号码可分配给多个员工共用
- 非共享号码同一时间只能分配给一个员工
- 共享号码的工单操作逻辑与非共享号码一致

### PHONE-004：号码操作与工单类型对应关系

| 操作 | 触发的状态变化 | 说明 |
|------|--------------|------|
| allocate（分配） | idle → active | 从闲置池分配给员工 |
| change_user（过户） | active → active | 更换使用人，号码不变 |
| change_number（换号） | active → active | 分配新号码，旧号码回池 |
| change_org（转移） | active → active | 变更归属部门 |
| reclaim（回收） | active/stopped → idle | 员工离职，号码回池 |
| trouble（停机） | active → stopped | 申请暂停 |
| trouble（复机） | stopped → active | 恢复使用 |
| surrender（拆机） | active/stopped/idle → cancelled | 归还运营商，不可逆 |

### PHONE-005：历史记录规则
- `phone_history` 记录所有号码状态变更，**永久保留**，无删除权限
- 每次状态变更均记录：变更前后状态、使用人、组织、操作人、操作时间、关联工单号

### PHONE-006：已拆机归档规则
- 号码拆机（surrender）后，从 `phone_number` 表中移除（或状态标记为 cancelled）
- 拆机信息写入 `phone_surrender_record` 表，**永久归档**
- 已拆机号码不参与后续任何业务操作和分摊

### PHONE-007：月度快照生成规则
- 每月 1 日自动生成上月快照（或手动触发）
- 快照纳入：`active` + `stopped`（可分配/在分摊范围内）
- 快照不纳入：`idle`（未分配，无需分摊）
- 快照纳入但标记不参与分摊：`cancelled`（已拆机号码记入历史，标记 `is_surrendered=1`）
- 快照字段：phone_number、user_id、extension_number、status、org_id、成本中心代码、月份
- 快照数据锁定后**不允许修改**

### PHONE-008：员工离职号码处理
- 管理员提交工单 → 运维执行 → 号码状态变为 `idle`
- 号码保留在 `phone_number` 表，可重新分配

## 三、工单规则

### WO-001：工单基本流程

```
[管理员创建 pending] → [运维接收 accepted] → [运维处理 processing]
→ [运维完成 completed] → [管理员归档 archived]
```

- 工单状态不可逆向跳转（completed 后只能归档）
- 工单创建后可直接取消（cancelled），需填写取消原因

### WO-002：无审批节点
- 工单流转无审批环节，直接由运维执行
- 工单结果即确认，无需二次审批

### WO-003：无超时规则
- 系统不设工单超时自动处理逻辑
- 超时提醒由人工或外部流程处理

### WO-004：通知规则
- 工单状态变更仅推送**系统消息**
- 不发送邮件、短信或其他外部通知
- 通知对象：申请人（Admin）、处理人（Ops）

### WO-005：批量工单规则
- 管理员可一次性提交多条号码的工单（单次最多 100 条）
- 批量工单按部门自动拆分为独立工单（`batch_id` 关联）
- 每条号码在拆分后的工单中对应一条 `work_order_item` 记录

### WO-006：跨组织工单
- **待确认**：Admin A 能否为 Admin B 所管辖部门的员工/号码提交工单？（目前按「不能跨部门」处理，待最终确认）

## 四、账单与分摊规则

### BILL-001：账单导入规则
- 仅支持 Excel（.xlsx）格式
- 导入时必须选择对应账单月份
- 账单数据进入 `bill_raw` 表永久保留，不允许删除
- **Excel 列名与系统字段映射关系：待确认**

### BILL-002：自动分摊逻辑
- 系统根据当月快照（phone_snapshot）匹配号码归属部门
- 账单金额分配到快照中记录的归属部门
- 匹配逻辑：`bill_raw.phone_number` → `phone_snapshot.phone_number`（指定月份）

### BILL-003：异常检测规则
- 系统对每条分摊明细进行异常检测，标记条件：
  - 金额与历史月均差异超过 ±50%（阈值可配置）
  - 号码当月无快照但有账单
  - 金额为零或异常大
- 异常数据自动标记 `anomaly_flag=1`，进入待审流程

### BILL-004：分摊确认流程

| 步骤 | 角色 | 确认内容 | 操作 |
|------|------|----------|------|
| 1 | 运维 | 导入账单 | 更新 import_status |
| 2 | 系统 | 自动匹配快照生成待确认分摊明细 | 写入 bill_allocation |
| 3 | 行政 | 确认部门归属正确 | admin_confirm_org |
| 4 | 行政 | 确认金额正确 | admin_confirm_amount |
| 5 | 财务 | 确认异常标记 | finance_confirm_anomaly |
| 6 | 财务 | 最终提交 | finance_confirm_submit |

- 任一步骤标记"错误/驳回"（wrong/rejected）：退回运维重新处理
- 全部确认后：分摊数据正式生效

### BILL-005：子公司对账规则
- 子公司财务查看本子公司及旗下各部门分摊汇总
- 汇总后向集团财务发起对账确认
- 确认方式：**待补充**（线上/线下）

## 五、发票分发规则

### INV-001：发票格式与上传
- 格式：PDF（每个发票独立文件）
- 上传方：集团总部财务（仅集团总部财务可操作）
- 上传时必须选择**账单月份**

### INV-002：发票命名与识别规则
- 文件命名约定：`{公司名称}_序号.PDF`
- **系统从 PDF 文件内容中自动提取公司名称**（OCR/文字提取）
- 不依赖文件名中的公司名称字符串（文件名仅作参考）
- OCR 提取后匹配组织架构中的公司名称，定位接收方

### INV-003：发票分发逻辑

| 发票来源公司 | 接收方 | 识别方式 |
|-------------|--------|---------|
| 集团总部 | 集团总部财务 | 公司名称="集团总部" |
| 子公司A | 子公司A财务 | 从 PDF 内容提取名称，匹配子公司组织 |
| ... | ... | ... |

### INV-004：无法识别的发票处理
- OCR 提取名称无法匹配任何子公司：状态标记为 `pending`
- 通知集团总部财务（系统消息）→ 人工指定接收方
- 人工指定后继续分发流程

### INV-005：发票状态流转

```
[上传 pending] → [识别中] → [distributed 分发成功] → [read 已读] → [confirmed 确认]
                        ↓
                   [pending 待人工确认]（无法识别时）
```

## 六、成本中心规则

### CC-001：维护权限
- 集团总部各部门的成本中心对照表：**由集团总部财务维护**
- 子公司成本中心：**待确认**（由谁维护？）
- 同一部门可对应多个成本中心（多对多关系）

### CC-002：变更管理
- 成本中心对照表变更需记录操作日志
- 历史映射关系在分摊快照中已锁定，不受后续修改影响

## 七、用户权限规则

### AUTH-001：角色定义

| 角色 | 说明 |
|------|------|
| Admin（行政） | 本部门及下级组织的管理，包括组织、员工、号码操作 |
| Ops（运维） | 全局电话运维：工单处理、账单导入 |
| Finance（财务） | 集团财务（成本中心、发票上传/分发、账单确认）；子公司财务（发票确认、对账） |
| Boss（老板） | 全局查看权限 |

### AUTH-002：登录方式
- 支持多方式登录：用户名+密码
- 支持 SSO 集成（钉钉/企业微信/飞书），具体对接：**待确认**

### AUTH-003：数据可见性
- Admin 只能看到管理范围内的数据
- Ops、Finance、Boss 按角色定义全局或财务数据可见范围
- 离职员工的历史数据（号码/分摊）永久保留，可查不可改

---

## 八、报表规则

### RPT-001：报表数据来源
- 所有报表数据基于月度快照（phone_snapshot）和分摊数据（bill_allocation）
- 报表数据均为历史快照，不实时变化

### RPT-002：报表权限
- 报表按角色和范围分发（Admin 只能看本部门报表）
- Boss 全局可看
